Mental model

The room needs a model of the work before a model of the tools.

Beginner-safe language first. Tool taxonomy later.

Memorable frame

The confident intern

  • Fast, well-read, enthusiastic, and confidently wrong without direction.
  • The human job is business judgment, architectural intent, and the final ship decision.
  • This framing reduces fear without overselling autonomy.
Phrase to keep
“Brilliant at following instructions. Terrible at inventing them.”
That line grounds the entire workshop: agents amplify disciplined direction; they do not replace it.

Thinking before typing

Research and framing are separate jobs from implementation.

A bad line of research becomes a lot of bad code quickly. Agentic work starts by making the target, the constraints, and the proof visible before the model touches the keyboard.

Step 1
Think
Clarify the goal, the quality bar, the evidence, and the rails.
Step 2
Type
Only then let the model execute within the defined loop.
Workshop shortcut
“Don’t give it a keyboard until you’ve given it a map.”

Quality bar

Vibe coding and disciplined agentic work are different operating modes.

Vibe coding Disciplined agentic work
Prompt into existence Define intent, scope, and proof first
One-shot output Multi-step loop with review, evidence, and iteration
“Seems fine” “I reviewed it, ran it, and can explain it”
Hard to repeat Easy to rerun, inspect, and improve

Context engineering

Curate what the model sees. Don’t reward it with clutter.

Copilot CLI makes the abstract idea concrete.

Working definition

Context engineering is the deliberate curation of what the model sees.

  • Best context is often proof of possibility: specs, examples, runbooks, screenshots.
  • The context window behaves like a whiteboard: clutter degrades reasoning quality.
  • The workshop’s own context sources are explicit: issue, OpenSpec, repo rules, plan, files, checkpoints.
Anchor lesson
Curate, don’t dump.
The quality jump rarely comes from “clever prompting.” It comes from narrowing the world the agent must reason over.

Copilot CLI topology

Two workspaces, one run.

Copilot CLI splits context into a working directory that Git owns and a session workspace that holds ephemeral plans, files, and checkpoints.

Layer 1
Working directory
Product code, tests, hooks, repo-wide instructions, shared conventions.
Layer 2
Session workspace
plan.md, files/, checkpoints, research outputs, resume state.
Practical rule
Team rules belong in the repo. Current-run thinking belongs outside Git.

Failure mode

Restart for poison. Compact for scale.

  • A poisoned session compounds bad assumptions; it does not heal through argument.
  • /compact reduces size in healthy long sessions. It does not erase wrong mental models.
  • The restart move is normal operating hygiene: summarize → exit → reopen → continue from the plan.
Symptoms
Repeated contradictions, fake constraints, or “fixes” that never converge.
Safe move
Fresh session, minimal summary, clean context, no emotional sunk-cost recovery.

Capability discipline

Skills teach how. MCP extends reach.

  • Skills are portable, on-demand workflow knowledge packs.
  • MCP adds tools, APIs, browsers, databases, and therefore more surface area.
  • The workshop framing stays simple: expertise → skill, new capability → MCP.
Use skills by default
Better token economics, stronger portability, narrower activation.
Enable MCP intentionally
Only when the workflow genuinely needs new reach into browsers, APIs, or systems.

Harness engineering

Good prompts do not make reliable systems. Loops, evidence, and review do.

This is where the proof chain becomes operational.

Core rhythm

Research → plan → implement → review

  • Every phase should produce an artifact, not just chat.
  • The phase boundary between planning and execution is part of the harness, not ceremony.
  • The workshop uses this rhythm to make OpenSpec-first delivery legible to mixed roles.
01
Research
Clarify the problem and gather proofs.
02
Plan
Constrain the path and success conditions.
03
Implement
Execute in isolated, reviewable steps.
04
Review
Check the diff, the behavior, and the evidence.

Role separation

Planner → generator → evaluator beats self-critique.

Complex work improves when planning, making, and checking stop blurring together. Copilot CLI gives this shape through the main agent, /fleet workers, and visible /tasks coordination.

Planner
Clarifies the path, dependencies, and success conditions.
Generator
Executes the scoped step with the right tools.
Evaluator
Compares outcome against proof, tests, review rules, and intent.

Reviewability

Fresh sessions, git diff, and reviewer respect are part of the harness.

  • Fresh sessions isolate reasoning and reduce context bleed.
  • Git diff is the local PR review between you and the agent.
  • Reviewer respect means no opaque, unreviewed AI dump gets promoted just because it compiled.

Cross-domain transfer

The three disciplines survive outside engineering.

  • Context engineering: policies, schemas, templates, calendars, approval rules.
  • Harness engineering: checks, permissions, audit trails, human approvals, rollback.
  • Orchestration: the workflow shape that moves work between tools, agents, and humans.

05

Agentic engineering

This is the synthesis, not the opener.

Only after context and harness make sense does autonomy become a practical conversation.

Human role shift

The human moves from typing every step to designing and reviewing the loop.

Agentic engineering is not a product. It is the discipline of giving the loop the right job, the right rules, and the right evidence — then deciding how much autonomy the task actually deserves.

Before
Manual execution, ad hoc prompting, vague acceptance, local intuition.
After
Intent setting, context curation, harness design, review, and progressive autonomy.

Autonomy ladder

Increase autonomy only when context, boundaries, and verification are stable.

0
Chat assist
Good for idea shaping and local edits.
1
Agent task
Clear win condition, scoped tools, reviewable output.
2
Multi-step loop
Research, plan, implement, review with visible artifacts.
3
Orchestrated workflow
Stable recurring work with queues, approvals, SLAs, or cross-functional routing.
Decision rule
If context is unstable or proof is fuzzy, don’t increase autonomy — tighten the harness first.
