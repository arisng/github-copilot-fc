# v2.10.0 Knowledge Pipeline Refactor — Design Rationale

## Why CURATE Mode Was Removed

The v2.9.0 pipeline had three Librarian modes: STAGE → PROMOTE → CURATE. CURATE served as an approval gate where the Librarian polled for APPROVE/SKIP signals from the human before promoting knowledge to the workspace wiki.

**Problems with CURATE:**

1. **Friction without value:** In practice, humans almost always approved extracted knowledge. The CURATE gate added a mandatory wait state that blocked the pipeline unnecessarily. Most knowledge items were uncontroversial (stable contracts, conventions, design decisions).

2. **Redundant with feedback loops:** Ralph-v2 already has a feedback-driven replanning system. If promoted knowledge was wrong, the human could provide feedback in the next iteration — a more natural and less disruptive correction mechanism than pre-promotion gating.

3. **Naming confusion:** Having both CURATE (a state) and APPROVE (a signal) created naming collisions documented in v2.9.0. The rename from `KNOWLEDGE_APPROVAL` → `CURATE` only partially addressed this.

4. **Pipeline simplification:** Removing CURATE collapsed the approval concern into a single SKIP check within PROMOTE, yielding a cleaner three-tier pipeline (EXTRACT → STAGE → PROMOTE) where each tier corresponds to a clear scope boundary (iteration → session → workspace).

## Why Auto-Promote with SKIP Opt-Out

The design inverts the approval model:

| Model | Default | Override |
|-------|---------|----------|
| v2.9.0 (CURATE) | Wait for APPROVE signal | SKIP to bypass |
| v2.10.0 (PROMOTE) | Auto-promote immediately | SKIP to opt-out |

**Rationale:**

- **Optimistic by default:** Knowledge extraction already filters for reusable, stable content. The extraction step itself is the quality gate. Adding a second gate (approval) duplicated the quality check.
- **Opt-out is lighter than opt-in:** The SKIP signal provides an escape hatch for the rare case where a human wants to prevent promotion. This matches the 80/20 pattern — optimize for the common case (promote) and provide an override for the exception (skip).
- **Reduced signal types:** Removing APPROVE reduced signal types from 6 to 5, simplifying the signal protocol. Every signal type adds complexity to every agent's checkpoint logic. Fewer signal types means simpler, more predictable agent behavior.
- **Feedback as the correction mechanism:** If promoted knowledge turns out to be wrong, the human provides feedback in the next iteration. This is already a well-exercised path in Ralph-v2's iteration model.

## Why the 5-Case Merge Algorithm

Both STAGE (iteration → session) and PROMOTE (session → workspace) use the same merge algorithm. This was a deliberate design choice.

### Per-File, Per-Category Merge

The merge operates within each Diátaxis category directory independently. A file in `reference/` never merges with a file in `explanation/`. This preserves Diátaxis classification integrity.

### The 5 Cases

1. **New file** — No matching filename in target. Straightforward copy. This is the most common case for first-time knowledge extraction.

2. **Same file, source newer** — Overwrites the target. This handles the case where knowledge was re-extracted in a later iteration with updated content. Newer-wins is the default conflict strategy.

3. **Same file, target newer** — Skips the source. This prevents accidental regression where an older extraction overwrites a newer one. Important for cross-iteration staging where `SOURCE_ITERATIONS` might include older iterations.

4. **Content overlap** — Different filenames but significant heading overlap (>50% H2/H3 heading match). Appends unique sections rather than creating near-duplicate files. This is a structural heuristic, not semantic — it trades false negatives (missed overlap → some duplication) for avoiding false positives (incorrect merge → data loss).

5. **Contradictory content** — Same heading with different content. Newer version wins entirely. This is the most aggressive resolution case, justified by the assumption that the most recent extraction reflects the most current state of the codebase.

### Why Newer-Wins (Timestamp-Based)

- **Deterministic:** No human judgment needed for conflict resolution. The merge can run fully automated.
- **Aligned with iteration model:** Later iterations inherently reflect more recent decisions. Knowledge extracted in iteration 5 supersedes knowledge from iteration 2.
- **Reversible via feedback:** If newer-wins produces wrong results, the human can provide feedback. The correction flows through the standard iteration cycle.

## Why Iteration-Scoped Knowledge

v2.9.0 wrote extracted knowledge directly to session-level `knowledge/`. v2.10.0 introduces `iterations/<N>/knowledge/` as an intermediate extraction target.

**Rationale:**

- **Isolation:** Each iteration's knowledge is self-contained. Re-running EXTRACT on iteration 3 doesn't affect iteration 2's knowledge.
- **Cherry-pick staging:** The `CHERRY_PICK` parameter enables selective staging — promoting only specific files from an iteration's knowledge. This requires iteration-scoped storage to provide a stable source.
- **Cross-iteration staging:** The `SOURCE_ITERATIONS` parameter enables staging knowledge from multiple previous iterations. This requires each iteration to have its own knowledge directory.
- **Audit trail:** Iteration-scoped knowledge provides clearer provenance — you can trace exactly which iteration produced which knowledge file, without relying solely on frontmatter metadata.

## Why Cherry-Pick and Cross-Iteration Staging

Two new STAGE mode parameters enable fine-grained control:

| Parameter | Use Case |
|-----------|----------|
| `CHERRY_PICK` | "Only stage these 2 files from iteration 3's knowledge" |
| `SOURCE_ITERATIONS` | "Stage knowledge from iterations 2, 3, and 5 (skip 4)" |

**Design motivation:**

- **Human batch control:** Humans can review iteration knowledge after EXTRACT and before STAGE, then selectively promote via cherry-pick. This replaces the pre-promotion approval gate with a more flexible pre-staging filter.
- **Catch-up staging:** If knowledge extraction was skipped or incomplete in earlier iterations, cross-iteration staging allows retroactive promotion without re-running EXTRACT.
- **Default is still automatic:** Without these parameters, STAGE defaults to staging all knowledge from the current iteration. The parameters are opt-in complexity.
