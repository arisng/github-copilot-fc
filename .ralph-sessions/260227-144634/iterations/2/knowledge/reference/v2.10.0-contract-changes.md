# v2.10.0 Knowledge Pipeline Refactor — Contract and API Changes

> Breaking: NOT backward-compatible with v2.9.0 sessions (session structure change + state machine change).

## Pipeline Rename

The Librarian's knowledge pipeline modes were renamed from a 3-mode system to a clearer 3-tier pipeline:

| v2.9.0 (Before) | v2.10.0 (After) | Purpose |
|------------------|------------------|---------|
| STAGE | EXTRACT | Scan iteration artifacts → write to `iterations/<N>/knowledge/` |
| PROMOTE | STAGE | Merge iteration knowledge → session `knowledge/` |
| CURATE | PROMOTE | Merge session knowledge → workspace `.docs/` |

**Key change:** CURATE mode was removed entirely. Its signal-polling responsibility (SKIP check) was absorbed by the new PROMOTE mode.

## Signal Type Changes

APPROVE signal was removed. Signal types reduced from 6 to 5:

| Signal | v2.9.0 | v2.10.0 | Scope |
|--------|--------|---------|-------|
| STEER | ✅ | ✅ | Universal |
| INFO | ✅ | ✅ | Universal |
| PAUSE | ✅ | ✅ | Universal |
| ABORT | ✅ | ✅ | Universal |
| SKIP | ✅ | ✅ | State-specific (PROMOTE mode within KNOWLEDGE_EXTRACTION) |
| APPROVE | ✅ | ❌ Removed | Was state-specific (CURATE mode) |

**SKIP ownership:** Consumed by the Librarian during PROMOTE mode within the `KNOWLEDGE_EXTRACTION` state. The Librarian polls `signals/inputs/` directly and archives on consume before transitioning.

## Frontmatter Field Renames

Knowledge file frontmatter fields were renamed to match the new pipeline terminology:

| v2.9.0 Field | v2.10.0 Field | Type |
|--------------|---------------|------|
| `approved` | `promoted` | boolean |
| `approved_at` | `promoted_at` | ISO8601 timestamp (nullable) |

## New Frontmatter Template

Every extracted knowledge file now includes three-state lifecycle tracking:

```yaml
---
category: tutorials | how-to | reference | explanation
source_session: <SESSION_ID>
source_iteration: <N>
source_artifacts:
  - iterations/<N>/tasks/task-X.md
  - iterations/<N>/reports/task-X-report.md
extracted_at: <ISO8601 timestamp>
staged: false
staged_at: null
promoted: false
promoted_at: null
---
```

## Outcome Value Changes

Librarian return values and Orchestrator contract outcomes were updated:

| v2.9.0 Outcome | v2.10.0 Outcome | Context |
|-----------------|------------------|---------|
| `approved` | `promoted` | Successful knowledge promotion |
| `skipped` | `skipped` | SKIP signal opt-out (unchanged) |
| `awaiting` | Removed | No longer needed (auto-promote is default) |
| `replanning` | Removed | Moved to orchestrator feedback detection |
| `curated` | Removed | CURATE mode no longer exists |

PROMOTE mode outcomes: `"promoted"`, `"skipped"`, or `"blocked"`.

## State Machine Changes

| Change | Detail |
|--------|--------|
| CURATE state removed | `KNOWLEDGE_EXTRACTION` now auto-sequences EXTRACT → STAGE → PROMOTE |
| Auto-sequence default | Orchestrator invokes all three Librarian modes in sequence within `KNOWLEDGE_EXTRACTION` state |
| SKIP opt-out | PROMOTE checks for SKIP signal before promoting; if found, outcome is `"skipped"` |
| 0-item short-circuit | If EXTRACT returns 0 items, Orchestrator skips STAGE and PROMOTE, transitions to COMPLETE |

## Session Structure: Iteration-Scoped Knowledge

New directory added to session structure:

```
iterations/<N>/knowledge/
  tutorials/
  how-to/
  reference/
  explanation/
  index.md
```

This is the extraction target for EXTRACT mode. Previously, knowledge was written directly to session-level `knowledge/`.

## Librarian Parameters

New optional parameters added to STAGE mode:

| Parameter | Mode | Purpose |
|-----------|------|---------|
| `SOURCE_ITERATIONS` | STAGE | List of iteration numbers to stage from (default: `[ITERATION]`) |
| `CHERRY_PICK` | STAGE | List of specific file paths to stage (stages only these files) |

## Merge Algorithm (5-Case)

Shared by STAGE and PROMOTE, operating per-file within each Diátaxis category:

| Case | Condition | Action |
|------|-----------|--------|
| New file | No matching filename in target | Copy directly |
| Same file, source newer | Source timestamp > target timestamp | Overwrite target |
| Same file, target newer | Target timestamp > source timestamp | Skip source file |
| Content overlap | Different filenames, same category, >50% H2/H3 heading overlap | Append unique sections |
| Contradictory content | Same heading with different content | Newer version wins entirely |

Content overlap detection uses a structural heuristic: H2/H3 heading comparison with >50% threshold.
