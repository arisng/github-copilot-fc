# Template Index and Reference Guide

This directory contains templates for each issue document type. Choose the right template based on your use case.

## Quick Reference Table

| **Template** | **Use When** | **Key Sections** | **Severity** | **Status Values** |
|---|---|---|---|---|
| **bug-report.md** | Something broke or isn't working | Problem, Root Cause, Solution, Prevention | Critical / High / Medium / Low / N/A | Resolved / In Progress / Investigating |
| **feature-plan.md** | Proposing a new feature or capability | Goal, Requirements, Implementation, Risks | Critical / High / Medium / Low / N/A | Draft / Proposed / In Progress / Accepted |
| **rfc.md** | Need team consensus on a design or proposal | Summary, Motivation, Detailed Design, Alternatives | Critical / High / Medium / Low / N/A | Open for Comment / Proposed / Accepted / In Progress |
| **adr.md** | Documenting a major decision after it's made | Context, Decision, Consequences | N/A | Proposed / Accepted / Superseded |
| **task.md** | Tracking discrete work or a project | Objective, Tasks, Acceptance Criteria | Critical / High / Medium / Low / N/A | Draft / Proposed / In Progress / Accepted |
| **retrospective.md** | Learning from an incident or project | Context, What Went Well, What Didn't, Lessons | N/A | Documented / Reviewed / Implemented |

## Common Frontmatter Fields

All templates use YAML frontmatter with metadata:

```yaml
---
date: YYYY-MM-DD                    # ISO 8601 date (required)
type: [Bug | Feature Plan | RFC | ADR | Task | Retrospective]  # Document type (required)
severity: Critical | High | Medium | Low | N/A  # Impact (optional, N/A for ADRs and retrospectives)
status: [varies by type]             # Current state (required)
author: Name <email>                 # Author (optional, recommended for ADRs)
tags: [list, of, tags]              # Semantic labels (optional)
related: [list, of, filenames]      # Related documents (optional)
---
```

## Workflow Example

**Scenario**: You found a bug in production and want to document it.

1. **Choose template**: Bug broke something â†’ BUG REPORT
2. **Copy template**: `cp templates/bug-report.md .issues/$(date +%y%m%d)_bug-slug.md`
3. **Fill in metadata**: Update `date`, `severity` (set to High if it's affecting users), `status` (Investigating while you work)
4. **Write content**: Follow the Problem â†’ Root Cause â†’ Solution structure
5. **Save and update status**: Once fixed, change status to Resolved
6. **(Optional) Index**: Run `python scripts/extract_issue_metadata.py` to regenerate `.issues/index.md`

## Tips for Writing Effective Documents

- **Be concise**: One page or less. Link to external docs if you need more detail.
- **Use examples**: Show code, configs, or data. Concrete beats abstract.
- **Link to source**: Reference code files, pull requests, or related issues by path or URL.
- **Act on decisions**: If you write an ADR or retrospective, ensure action items have owners.
- **Update status regularly**: Readers check the status field to understand the lifecycle.
